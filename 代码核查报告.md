## 三、代码质量评估

### 3.1 HTML 结构

**优点：**
- 语义化 HTML5 标签使用规范（`<header>`, `<nav>`, `<section>`, `<footer>`）
- `data-i18n` 属性作为国际化标记清晰有序
- 响应式设计已实现（CSS 中应含 media query）
- `aria-expanded` 等无障碍属性已部分实现

**问题：**
- HTML 页面间存在大量重复的 Header/Footer 结构，未使用模板引擎或组件化方案，维护成本高
- 首页 393 行中约 200 行为静态HTML结构，不利于 SEO 和加载优化
- 部分 `<img>` 标签缺少有意义的 `alt` 描述（尽管产品卡片渲染中有 `alt`）

### 3.2 CSS 质量

**优点：**
- 使用了 CSS 自定义属性（CSS Variables）进行主题管理
- 使用现代布局方式（Flexbox / Grid）
- 暗色主题配色统一专业

**问题：**
- `style.css` 高达 80KB，单文件未模块化，不便维护
- 未使用 CSS 预处理器（Sass/Less），无自动前缀（Autoprefixer）
- 未发现 Critical CSS 优化策略（首屏加载可能较慢）
- 部分样式通过 JS 内联设置（如 `main.js` 中的 `style.display`），应优先使用 CSS class 切换

### 3.3 JavaScript 质量

**优点：**
- `main.js` 粒子动画 Canvas 实现精巧，performance 优化良好（resize/scroll 均做节流处理）
- `data-loader.js` 使用 IIFE 模式封装，避免全局污染
- `i18n.js` 使用 localStorage 持久化语言偏好
- 产品筛选动画过渡流畅

**问题：**
- **i18n.js 存在严重数据错误**（详见 §4.1），直接影响线上多语言功能
- `data-loader.js` 中 `fetchJSON` 无超时控制，网络异常时可能长时间挂起
- `main.js` 中联系表单提交用 `setTimeout` 模拟异步，实为假提交（未真正调用 API）
- 部分 JS 直接操作 `style` 属性，违反关注点分离原则

### 3.4 响应式设计

- 页面使用了 `viewport` meta 标签
- 导航栏支持移动端汉堡菜单（`menuToggle` + `mainNav`）
- admin.html 侧边栏为固定定位，移动端欠缺适配
- 产品卡片使用 Grid 布局，应可自适应

### 3.5 SEO 友好度

**已有措施：**
- 每个页面有独立的 `<title>`（通过 i18n 的 `page_title_*` 键）
- 语义化 HTML 结构

**缺失措施：**
- 无 `<meta name="description">` 动态设置
- 无 Open Graph / Twitter Card 标签
- 无 canonical URL
- 无 hreflang 标签（多语言站点的关键 SEO 要素）
- 无 XML Sitemap / robots.txt
- 无结构化数据（Schema.org JSON-LD）

### 3.6 加载性能

**潜在问题：**
- 图片资源约 9MB，未使用 WebP 格式，未实现懒加载
- CSS 80KB 未压缩 / 未拆分
- JS 文件无代码分割，无 Tree Shaking
- 未使用 CDN 加速静态资源
- 无 Service Worker / PWA 支持

## 四、发现的问题与风险

### 4.1 🔴 严重问题

#### (1) i18n.js 翻译数据错误 — 英文版混入大量中文

**影响范围：** 当用户切换到英文时，大量内容会显示中文而非英文。

**问题详情：**

i18n.js 的 `en` 翻译对象中，以下键值仍然为**中文**（未翻译）：

```
en:
  - page_title_index → 中文
  - page_title_products → 中文
  - page_title_about → 中文
  - page_title_services → 中文
  - page_title_blog → 中文
  - page_title_contact → 中文
  - back_to_top → "返回顶部"
  - contact_map_text → "中国浙江省宁波市"
  - contact_map_sub → "国际商务中心"
  - btn_details → "详情"
  - btn_inquiry → "询盘"
  - blog1_date ~ blog9_date → 部分中文
  - about_tl_2008 ~ about_tl_2025 → 部分中文
  - contact_country_us ~ contact_country_other → 全部中文
```

同时存在**重复键定义**：`blog1_date`, `blog2_date`, `blog3_date`, `blog4_date` ~ `blog9_date`, `about_tl_*`, `contact_country_*`, `page_title_*`, `btn_details`, `btn_inquiry`, `back_to_top` 等在 en 翻译对象中先后定义了两次（一次中文值，一次英文值），后定义会覆盖先定义的，最终结果取决于覆盖顺序，极不稳定。

**修复建议：** 删除重复键，将所有中文值替换为对应英文翻译。

#### (2) server.js 安全硬编码

```javascript
const JWT_SECRET = 'waterbearintl_jwt_secret_key_2026';  // 第 8 行
// 默认账号 admin / admin123 硬编码在用户初始化逻辑中
```

**风险：** JWT_SECRET 硬编码意味着任何拿到源码的人都能伪造 JWT Token；默认管理员密码过弱。

**修复建议：** 使用环境变量 `process.env.JWT_SECRET`；首次启动强制修改默认密码。

#### (3) 联系表单假提交

`main.js` 第 120-130 行：

```javascript
setTimeout(() => {
  showNotification(..., 'success');
  contactForm.reset();
  ...
}, 1200);
```

**风险：** 表单提交后始终显示成功，但并未实际调用 `GlobexData.submitContactForm()` 发送数据。用户询盘会丢失。

**修复建议：** 将 `setTimeout` 替换为 `await GlobexData.submitContactForm(formData)` 真实调用。

### 4.2 🟡 中等问题

#### (4) 后端使用 JSON 文件存储

- 无并发保护（多请求同时写文件会导致数据损坏）
- 数据量增长后性能急剧下降
- 无法进行复杂查询

**修复建议：** 迁移到 SQLite（轻量方案）或 PostgreSQL（生产方案）。

#### (5) 前端无错误边界

- API 调用失败仅有 `console.warn`，用户看不到任何错误提示
- 产品数据加载失败时页面显示空白卡片

#### (6) 无输入校验

- server.js 中未对用户输入做 XSS/SQL 注入防护
- 文件上传未限制类型和大小（可能被上传恶意文件）

#### (7) 无访问速率限制

- API 端点无 rate limiting，容易被滥用或 DDoS

#### (8) CSS 文件过大

- 80KB 单文件包含所有页面样式，影响首屏加载

### 4.3 🟢 轻微问题

- `.gitignore` 中可能未包含 `data/` 和 `backups/` 目录（需确认）
- 无 ESLint / Prettier 配置
- 无前端自动化构建工具（Webpack/Vite）
- 页面间导航切换无加载过渡效果
- admin.html 中大量硬编码的中文字符串，未使用 i18n
